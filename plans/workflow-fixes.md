# Workflow Fix Plan

## Wave 1 — No dependencies between tasks

### Task 1: [Critical] Add a commit step before PR submission

**Problem:** The workflow makes file changes (implement, refactor) but never commits them to the branch. When `submit_pr` runs, `gh pr create` fails because there are no commits between `main` and the feature branch.

**Scope:**
- `workflow/workflows/issue.py` — add a new `commit_changes` Stage between `refactor` (stage 7) and `gate_docs` (stage 8). Make it a mechanical stage that runs `git add -A && git commit -m <message>` using the context's summary from the implement stage.
- `workflow/tools/git.py` — add a `make_commit_tool()` factory function (or handle it mechanically).
- `tests/workflow/test_issue_workflow.py` — test that `build_issue_workflow()` includes the commit stage at the right position.
- `tests/workflow/test_git_tools.py` — test the new commit tool/function.
- The commit message should come from context (e.g. the implement stage's summary, or a fixed convention like the issue title).

**Key files to read first:**
- `workflow/workflows/issue.py` (lines 92–100: `_make_add_label_fn` pattern for mechanical stages)
- `workflow/tools/git.py` (existing tool factories)
- `workflow/context.py` (summaries dict holds prior stage summaries)
- `tests/workflow/test_issue_workflow.py` (existing tests)
- `tests/workflow/test_git_tools.py` (existing tests)

**Constraints:**
- Follow existing patterns (mechanical stage, factory functions, `shell()` with `shlex.quote`).
- Use `asyncio` timeouts (project convention).
- All pytest tests need timeouts (project convention from AGENTS.md).
- Don't swallow exceptions.
- Comment every block of code.

### Task 3: [Medium] Fix `stage_complete` outcome validation

**Problem:** The LLM repeatedly passes descriptive prose (e.g. `"Created and checked out branch..."`, `"NO"`, `"All 5 tests pass..."`) instead of one of the four valid enum values (`proceed`, `need_clarification`, `scope_changed`, `blocked`). The tool already returns an error string and the StageRunner falls back to `proceed`, but this wastes LLM turns.

**Root cause analysis:** The `stage_complete` tool function's docstring tells the LLM the valid values, but the tool schema generated by `tool_to_schema()` (in `corvidae/tool.py`) only sees `outcome: str` — it doesn't emit a JSON schema `enum` constraint. The LLM sees a free-text string parameter, not a constrained choice.

**Scope:**
- `workflow/tools/stage_complete.py` — the `stage_complete` inner function's `outcome` parameter needs to carry schema-level constraint info so `tool_to_schema()` can emit an `enum`. Options:
  1. Change the type annotation to `Literal["proceed", "need_clarification", "scope_changed", "blocked"]` — `tool_to_schema()` uses pydantic's `create_model()`, which should produce a JSON schema with `enum` from `Literal`.
  2. Or add an explicit schema override in `Tool.from_function()` / `tool_to_schema()`.
- Verify by reading `corvidae/tool.py:tool_to_schema()` to confirm `Literal` types produce `enum` in the schema. If they don't, add that handling to `tool_to_schema()`.
- `tests/workflow/test_stage_complete.py` — add test that the tool schema includes `enum` for `outcome`.
- Also improve the error message returned to the LLM when it fails — make it even more explicit ("You passed a sentence. Pass EXACTLY one of: ...").
- Consider: does `tool_to_schema()` strip `enum` when it strips `title`? Check this.

**Key files to read first:**
- `workflow/tools/stage_complete.py` (full file — the tool function and holder)
- `corvidae/tool.py` (`tool_to_schema()` function, ~line 50–60 — how schemas are generated)
- `tests/workflow/test_stage_complete.py` (existing tests)
- `tests/test_dispatch_tool_call.py` or `tests/test_tools.py` (schema generation tests)

**Constraints:**
- Don't change StageOutcome enum values or add new ones.
- The fix must make the LLM *see* the enum constraint in the tool schema it receives, not just in the docstring.
- Backward compatible — existing tests must still pass.

### Task 6: [Medium] Route logs to file, human-focused output to STDOUT

**Problem:** Currently `workflow/__main__.py` configures `logging.basicConfig()` which dumps all log lines (INFO-level, machine-formatted) to STDOUT. A user sees a wall of `2026-05-01 10:26:20,007 INFO __main__: Running workflow...` instead of a clean progress display.

**Scope:**
- `workflow/__main__.py` — replace `logging.basicConfig()` with proper dual-handler logging:
  - **File handler**: Write full DEBUG-level structured logs to a log file (e.g. `workflow.log` or a timestamped file in `.corvidae/logs/`). Use the existing `StructuredFormatter` from `corvidae/logging.py`.
  - **STDOUT handler**: Write human-readable stage progress. Format: stage name, one-line status, result. Show errors clearly. Not INFO-level internal logging.
- Create a new `workflow/display.py` module (or similar) that provides a simple progress display abstraction:
  - `display_stage_start(name)` — print "▸ stage_name"
  - `display_stage_complete(name, outcome)` — print "✓ stage_name (proceed)" or "✗ stage_name (blocked)"
  - `display_error(message)` — print errors clearly
  - `display_workflow_complete()` — final summary
- Hook the display into the engine: `workflow/engine.py` currently logs `Starting stage: X` and `Stage complete: X (outcome=Y)`. Add display calls alongside (or replace the logging for these specific events).
- `tests/workflow/test_cli.py` — test the display output.
- Don't import heavy TUI libraries — keep it plain `print()` with unicode symbols for now.

**Key files to read first:**
- `workflow/__main__.py` (current logging setup)
- `workflow/engine.py` (where stage start/complete events happen — lines 64–65 and 89–90)
- `corvidae/logging.py` (StructuredFormatter, configure_logging — reuse patterns)
- `tests/workflow/test_cli.py` (existing CLI tests)

**Constraints:**
- Don't change the daemon's logging (`corvidae/logging.py`) — only the workflow's logging setup.
- The log file should always capture everything (DEBUG level).
- STDOUT should be concise — one line per stage, not one line per tool call.

## Wave 2 — Depends on Wave 1

### Task 2: [High] Add recovery when `submit_pr` fails ✅ DONE

**Problem:** When `submit_pr` fails (outcome=blocked), the workflow tries to escape to `implement` but the engine rejects it because `submit_pr` doesn't declare `implement` as an escape target. The workflow dead-ends with no recovery, no error summary, and no cleanup.

**Depends on:** Task 1 (commit step) — the primary failure mode was "no commits". Once Task 1 adds a commit step, this specific failure should not recur. But the recovery mechanism is still needed for other PR failures (auth issues, network errors, branch conflicts).

**Scope:**
- `workflow/workflows/issue.py` — add escape_targets to the `submit_pr` stage: `{StageOutcome.BLOCKED: "commit_changes"}` (or `"implement"` if that's more appropriate).
- `workflow/engine.py` — when the engine stops due to a blocked/error outcome, print a clear summary to STDOUT (requires Task 6's display module):
  - Which stage failed and why
  - What the LLM's summary was
  - Suggested next steps
  - Branch name if one was created (for manual cleanup)
- Consider: should the engine attempt automatic retry (re-run the failed stage once), or just report and stop? Probably report and stop for now — the human can decide.
- `tests/workflow/test_engine.py` — add test for blocked-without-escape-target producing a clear end state (not a silent stop).
- `tests/workflow/test_engine.py` — add test for blocked-with-valid-escape-target jumping back.

**Key files to read first:**
- `workflow/engine.py` (the `else` branch at line ~101 — where non-PROCEED without escape target stops the workflow; and line ~87 — where invalid escape targets stop)
- `workflow/workflows/issue.py` (submit_pr stage definition)
- `tests/workflow/test_engine.py` (existing engine tests)

**Constraints:**
- Don't add infinite retry loops.
- Blocked outcome should always produce user-visible output (not just log lines).
- Escape targets must be explicitly declared — don't auto-escape to arbitrary stages.

**Documentation:**
- `workflow/README.md` — update the stage list in "Example Workflow: GitHub Issue Implementation" to show the new `commit_changes` stage (already updated in Wave 1) and note that `submit_pr` now has an escape target back to `commit_changes`. Update the "What's built" section to mention recovery from blocked stages.
- `plans/workflow-fixes.md` — mark Task 2 as done.
- No changes needed to `docs/` (those are about the daemon, not the workflow).

### ~~Task 4: [Medium] Fix gate stages to accept boolean outcomes properly~~ ✅

**Problem:** `gate_docs` is designed to check if docs need updating. The LLM is instructed to start its summary with "YES" or "NO", and `_docs_should_run()` checks the summary string. But the LLM passed `"NO"` as the *outcome* of `stage_complete` instead of the *summary*, triggering the invalid-outcome warning. The LLM is confused about where to put the YES/NO signal.

**Depends on:** Task 3 (outcome validation) — if the schema now has an `enum`, the LLM will be forced to pass a valid outcome. But the gate design itself needs fixing.

**Scope:**
- `workflow/workflows/issue.py` — redesign the gate_docs stage. Options:
  1. **Best**: Make `gate_docs` a mechanical stage that checks the diffs programmatically (did any doc files change? did any public API change?). No LLM needed.
  2. **Alternative**: Keep it LLM-driven but fix the instruction — tell the LLM to use `outcome=proceed` always, and put YES/NO in the summary. Fix the `_docs_should_run` function to read from `context.facts["gate_docs_decision"]` set by a dedicated tool, not from the summary string.
  3. **Simplest fix**: Just fix the instruction prompt to be unambiguous about where the YES/NO goes.
- Whichever approach, update `_docs_should_run()` to be robust.
- `tests/workflow/test_issue_workflow.py` — test that `_docs_should_run` works correctly for YES, NO, and edge cases.
- Also check `gate_ready` — it has a similar pattern but doesn't have the same bug (yet). Make its instructions equally clear.

**Key files to read first:**
- `workflow/workflows/issue.py` (lines ~117–130: gate_docs stage and `_docs_should_run`)
- `tests/workflow/test_issue_workflow.py` (existing tests)

**Constraints:**
- The gate_docs decision must be deterministic and testable, not dependent on LLM free-text parsing.
- Prefer mechanical/deterministic approaches per the README's "Prefer deterministic code" principle.

**Documentation:**
- `workflow/README.md` — update the description of the `gate_docs` stage in the example workflow section to reflect the new mechanical implementation. Remove any mention of YES/NO summary convention. Note that `gate_ready` instructions were also clarified.
- `plans/workflow-fixes.md` — mark Task 4 as done.
- No changes needed to `docs/`.

### Task 7: [Medium] Enrich log file with LLM/tool call detail ✅ DONE

The structured extra fields (tool name, arg_keys, result_length) are only visible when using `StructuredFormatter`. For workflow debugging, the log file needs to include the tool name, truncated arguments, and truncated results at INFO level.

**Depends on:** Task 6 (log file setup) — needs the log file to exist.

- The existing `corvidae/tool.py` and `corvidae/turn.py` logging is fine for the daemon — don't change it. Instead, add workflow-specific logging in `stage.py` that uses the `AgentTurnResult` and `ToolCallRecord` data.
- Option: promote the `tool_records` list into the stage-level logging so the log file shows a compact summary after each stage completes (e.g. "Stage implement completed: 8 tool calls (2 write_source, 3 run_command, 1 read_source, 1 stage_complete, 1 error)").
- `tests/workflow/test_stage_runner.py` — verify logging output.

**Key files to read first:**

### Task 5: [Low] Add a fast path for trivial tasks ✅ DONE

**Problem:** A one-line README change goes through the full TDD cycle (write_tests → implement → refactor), creating 5 test files and 1211 test runs. This is overkill for docs-only changes.

**Depends on:** Task 8 (structured artifacts from early stages) — needs the task classification to inform stage skipping. Could also be done independently with a simpler approach.

**Scope:**
- Option A (simple): Add a `should_run` predicate to `write_tests` and `refactor` stages that checks the issue body for keywords indicating a docs-only change. Fragile but quick.
- Option B (better, depends on Task 8): Use the structured plan artifact from `comment_on_issue` to determine if tests/refactoring are needed. The plan would include a `task_type` field (e.g. `docs_only`, `bug_fix`, `feature`, `refactor`).
- `workflow/workflows/issue.py` — add `should_run` lambdas that check `context.facts["task_type"]` or similar.
- `tests/workflow/test_issue_workflow.py` — test that trivial issues skip write_tests/refactor.
- Consider: should `gate_ready` also make this determination? Or should a new early stage classify the task?

**Key files to read first:**
- `workflow/workflows/issue.py` (stage definitions, `_docs_should_run` as pattern)
- `workflow/stage.py` (`should_run` field and how it's checked in `engine.py`)
- `workflow/engine.py` (line ~67: where `should_run` is checked)

**Constraints:**
- Don't remove stages — just skip them conditionally.
- The classification must be deterministic once decided (store in context.facts).
- The commit step (Task 1) must still run even on the fast path.

**Documentation:**

### Task 8: [Medium] Have early stages produce structured artifacts for later stages ✅ DONE

**Problem:** Each stage makes independent LLM calls. The `comment_on_issue` stage analyzes the issue and writes a plan, but later stages (create_branch, write_tests, implement) re-analyze the same issue from scratch. The branch name, file list, and task classification are computed redundantly.

**Depends on:** Tasks 3 and 4 (clean outcome/summary handling) — the structured artifact needs to be stored in `context.facts`, not in the free-text summary.

**Scope:**
- `workflow/workflows/issue.py` — redesign `comment_on_issue` to produce a structured artifact alongside the comment. Options:
  1. Add a `record_plan` tool that the LLM calls to save structured data (branch_name, files_to_change, task_type, needs_tests, needs_docs) to `context.facts`.
  2. Parse the LLM's summary into structured data (fragile).
  3. Add a new `classify_task` mechanical+LLM stage after `comment_on_issue` that produces the artifact.
- `workflow/tools/` — add a `make_record_plan_tool()` factory that writes to `context.facts["plan"]`.
- Later stages consume the plan:
  - `create_branch`: use `context.facts["plan"]["branch_name"]` instead of asking the LLM.
  - `write_tests`: skip if `context.facts["plan"]["needs_tests"] == False`.
  - `gate_docs`: use `context.facts["plan"]["needs_docs"]` instead of the YES/NO hack.
  - `submit_pr`: use the plan for PR title/body.
- `tests/workflow/test_issue_workflow.py` — test plan recording and consumption.
- `tests/workflow/test_context.py` — test that facts dict works as expected.

**Key files to read first:**
- `workflow/workflows/issue.py` (full file — all stage definitions)
- `workflow/context.py` (facts dict and build_context_string)
- `workflow/tools/github.py` (pattern for tool factories)
- `tests/workflow/test_context.py` (existing context tests)

**Constraints:**
- The plan tool should validate required fields.
- The plan should be a dict stored in `context.facts`, not a new mechanism.
- Existing stages must still work if the plan is missing (graceful degradation).
- Don't over-engineer — start with the minimal fields that unblock later stages.

**Documentation:**
- `workflow/README.md` — update the "Example Workflow" section to describe the new plan artifact. Add a "Plan artifact" subsection under "What's built" listing the fields (`branch_name`, `task_type`, `needs_tests`, `needs_docs`, `files_to_change`) and explaining how stages consume it. Update the `comment_on_issue` stage description to note it also calls `record_plan`. Update later stage descriptions (create_branch, gate_docs, submit_pr) to note they read from the plan when available.
- `plans/workflow-fixes.md` — mark Task 8 as done.
- No changes needed to `docs/`.
